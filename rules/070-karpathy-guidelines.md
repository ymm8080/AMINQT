# 070 — Karpathy LLM 编码行为指南 (12 Rules)

> **来源**: Karpathy (Jan 2026) + @mnilax 扩展 (May 2026, 30 个代码库, 6 周)
> **效果**: 无规则 41% 错误率 → 4 条 11% → **12 条 3%**
> **适用范围**: 所有 AI 生成代码的行为约束，始终生效

## 核心 4 条 (Karpathy via Forrest Chang)

### 1. 编码前先思考
**不要假设。不要隐藏困惑。暴露权衡。**

- 显式声明假设。不确定就问。
- 存在多种解释时，全部列出——不要默默选一个。
- 有更简单的方案时，直说。该反驳就反驳。
- 不清楚就停下。指出困惑点。问。

> [量化] 对策略逻辑有歧义时（如"选股"是选 Top 5 还是 Top 10？），必须确认后再写代码。

### 2. 简洁优先
**解决问题的最少代码。不做推测性开发。**

- 不加用户没要求的功能。
- 单次使用的代码不做抽象。
- 不加未要求的"灵活性"或"可配置性"。
- 不为不可能发生的场景写错误处理。
- 200 行能压到 50 行，就重写。

> [量化] factor_engine 不要预先实现 50 个指标。PROMPT_CONTENT 要求的 MACD/KDJ/BOLL/RSI + 衍生特征就够。多余指标增加过拟合风险。

**检验**: 资深工程师会说这太复杂吗？是 → 简化。

### 3. 外科手术式修改
**只动该动的。只清理自己制造的混乱。**

编辑现有代码时：
- 不"改进"相邻代码、注释、格式。
- 不重构没坏的东西。
- 匹配现有风格，即使你有不同偏好。
- 发现无关死代码 → 提及，不删除。

你的修改产生的孤儿：
- 删除**你的修改导致**的无用 import/变量/函数。
- 不删除**预先存在**的死代码（除非被要求）。

> [量化] 修改 `factor_engine.py` 时，不要顺手重构 `data_loader.py`。每个 Phase 只改本 Phase 的代码。

**检验**: 每个改动行都能追溯到用户的请求。

### 4. 目标驱动执行
**定义成功标准。循环直到验证通过。**

把任务转化为可验证目标：
- "加验证" → "为无效输入写测试，然后让它们通过"
- "修 bug" → "写一个复现 bug 的测试，然后让它通过"
- "重构 X" → "确保前后测试都通过"

多步任务：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```

> [量化] ↔ `verify-before-done` 技能。Phase 1 完成 = `data/raw/` 出现 5 个 CSV + 列名正确。Phase 2 完成 = X.shape[2] ≥ 25 + 无 NaN。

**关键洞察**: LLM 擅长循环直到达成目标。不要告诉它怎么做，给它成功标准。

## 扩展 8 条 (@mnilax, May 2026)

### 5. LLM 只做判断，确定性逻辑用代码
**确定性决策属于确定性代码。**

- LLM 用于：分类、起草、摘要、提取
- LLM **不用于**：路由、重试、状态码处理、确定性转换

> [量化] ↔ 铁律三 LLM 交易禁令。LLM 不参与下单决策、风控判断、T+1 检查。这些是确定性逻辑，必须用代码实现。

### 6. 硬性 Token 预算，无例外
**每个循环都可能变成 50,000 token 的上下文倾泻。**

- 单任务: ~4,000 tokens | 单会话: ~30,000 tokens
- 接近预算时 → **摘要后重新开始**
- **暴露超额 > 静默超限**

> [量化] 全市场选股遍历可能产生大量中间输出。控制 token，避免上下文爆炸导致 AI 忘记铁律。

### 7. 暴露冲突，不要平均
**代码库各部分不一致时，不要混搭。**

两种模式矛盾时：选一个（更新/测试过的那个），解释原因，标记另一个待清理。

> [量化] akshare 和 iFinD 的列名不一致 → 适配器统一映射，不在核心代码中混用中英文列名。参见 `rules/020-data-format-mapping.md`。

**"平均"代码满足两个规则 = 最差的代码。**

### 8. 写之前先读
**在给代码添加内容前，先理解相邻代码。**

添加代码前：读文件的导出、直接调用者、共享工具。
不理解现有代码为什么这样组织 → **先问再写**。

> [量化] 修改 `risk_filter.py` 前，先读 `config/settings.py` 中的 `MAX_ACCOUNT_DRAWDOWN_PCT` 等配置项，理解风控阈值从哪来。

### 9. 测试验证意图，不只验证行为
**通过的测试如果不验证有用的东西 = 失败。**

每个测试必须编码**为什么**这个行为重要，不只**是什么**。
写不出在业务逻辑变更时会失败的测试 → **函数设计有问题**。

> [量化] `test_factor_engine.py` 不应只测 "X.shape == (N, 20, 30)"，还应测"第 t 天的指标不依赖 t 以后的数据"（未来函数防护）。参见 `rules/010-future-function-prevention.md` §验证方法。

### 10. 每个重要步骤后检查点
**不要从一个你无法描述的状态继续。**

每步后：总结做了什么、验证了什么、剩下什么。
**失去跟踪 → 停下，重新陈述。**

> [量化] Phase 1 → 验证 CSV 存在 → Phase 2 → 验证 X.shape → Phase 3 → 验证 Loss < 0.01。每个 Phase 完成后做检查点，不跳步。

### 11. 匹配代码库惯例，即使你不同意
**成熟代码库中惯例胜过新意。**

- 代码库用 snake_case 你偏好 camelCase → **snake_case**
- 代码库用 logging 你偏好 print → **logging**
- 代码库内，**一致性 > 品味**。暴露分歧，不默默分叉。

> [量化] 项目已有 `# -*- coding: utf-8 -*-` 文件头惯例、Google 风格 docstring 惯例、`logging` 惯例。新代码必须遵守，不可"改进"。参见 `rules/050-code-conventions.md`。

### 12. 失败要大声
**无法确定时，明确说出来。**

- "迁移完成"在有记录被静默跳过时 = **错误**
- "测试通过"在跳过了某些测试时 = **错误**
- **默认暴露不确定性，不隐藏。**

> [量化] ↔ 铁律二 资金安全。实盘下单时，任何异常、跳过、降级都必须明确记录。审计日志 append-only。不静默跳过风控检查。参见 `rules/030-risk-filter-hard-constraints.md`。
