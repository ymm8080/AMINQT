# Skill: careful (关键系统双重检查)

> **来源**: EWM Robot 项目 `careful.md`
> **用途**: 关键系统代码的双重检查，确保正确性优先于速度
> **触发**: 实盘交易代码、金融计算、风控逻辑、模型推理、数据迁移

## 理念

**量两次，切一次。关键系统中错误代价高昂。**

> [量化] 交易代码的 bug = 直接亏钱。风控代码的 bug = 灾难。
> 这里"偏执是美德"。

## 适用场景

✅ **实盘交易代码** — executor 层、下单逻辑
✅ **风控代码** — risk_filter、熔断逻辑
✅ **金融计算** — 收益率、回撤、仓位计算
✅ **模型推理** — 预测得分影响选股
✅ **数据迁移** — CSV 格式变更、列名映射
✅ **配置变更** — 风控阈值、执行模式切换

## 检查清单

### 编码前
- [ ] 完全理解需求？
- [ ] 识别了所有边界情况？
- [ ] 有隐藏假设？
- [ ] 可能出什么错？（列出前 3 个失败模式）
- [ ] 有现有代码可参考？

### 编码中
- [ ] 验证了**所有**输入？
- [ ] 处理了**所有**错误情况？
- [ ] 有竞态条件？
- [ ] 线程安全？
- [ ] 资源泄漏？（文件句柄、内存）

### 编码后
- [ ] 测了正常路径？
- [ ] 测了错误路径？
- [ ] 测了边界情况？（空、null、最大、最小、除零）
- [ ] 意外输入优雅处理？
- [ ] 错误信息有帮助且安全？（不泄露凭据）

## 量化场景验证技术

### 1. 输入验证矩阵
```
每个参数测试：
- 有效值 ✓
- None
- 空字符串/空数组
- 最大长度/大小
- 无效类型
- 特殊字符
- 除零场景
- NaN 值
```

### 2. 风控状态机分析
```
列出所有可能状态：
- 正常状态（回撤 < 3%）
- 接近阈值（回撤 2.5%-3%）
- 熔断状态（回撤 > 3%）
- 人工恢复中

验证：每个状态都有到完成或错误的路径。
验证：熔断后不得自动恢复（铁律二）。
```

### 3. 交易数据流追踪
```
每笔交易数据：
1. 从哪来？（模型推理输出）
2. 怎么变换？（风控过滤）
3. 存到哪？（审计日志）
4. 谁能访问？
5. 何时删除？（审计日志保留 180 天后归档）
6. 每步验证什么？（风控阈值、T+1）
```

### 4. 错误传播
```
每个函数调用：
- 可能抛什么异常？
- 捕获了吗？
- 正确处理了吗？
- 静默吞异常了吗？（禁止！铁律六）
- 异常传播到正确层级了吗？
```

## 红旗（停止并审查）

🚩 修改交易执行器代码 → 必须双重验证
🚩 修改风控阈值 → 必须验证不降低安全
🚩 修改收益/回撤计算 → 必须用测试用例验证
🚩 删除数据 → 确定能恢复？
🚩 修改执行模式 (manual→auto) → 必须完整测试

## 黄金规则

1. **偏执是美德** — 假设一切都会失败
2. **不信任输入** — 每个边界都验证
3. **安全失败** — 错误不应损坏数据或资金
4. **记录一切** — 但不记录敏感数据
5. **先模拟后实盘** — 永远不在实盘直接测试
6. **有回滚计划** — 做之前知道怎么撤销
7. **二次审查** — 交易代码必须 code review
