# Skill: grill-me (编码前追问)

> **来源**: EWM Robot 项目 `grill-me.md`
> **用途**: 实现前先追问，确保理解正确，防止误解浪费精力
> **触发**: 新功能开发、重构现有代码、需求不明确、请求看起来异常时

## 编码前必问

1. **实际要解决什么问题？**（不是用户说了什么，而是真正需要什么）
2. **边界情况是什么？**（可能出什么错？）
3. **有更简单的方案吗？**（是否过度设计？）
4. **后果是什么？**（性能、安全、可维护性）

## 量化场景追问示例

❌ **坏**: 用户说"加个选股接口" → 直接写代码

✅ **好**: 用户说"加个选股接口" → 追问：
- 选 Top 5 还是 Top 10？
- 从内存缓存还是实时下载？
- 返回 JSON 还是直接下单？
- MANUAL 模式弹窗还是 AUTO 直接执行？
- 需要风控过滤吗？（必须！）
- 响应时间要求？（< 5 秒？）

## 流程

1. 仔细读请求
2. 识别假设
3. 问澄清问题
4. 提出替代方案（如适用）
5. 等用户确认后再继续

## 与其他技能配合

```
grill-me（追问需求）
  → superpowers（编码前检查）
    → careful（关键代码双重检查）
      → verify-before-done（完成后验证）
```

## 与文档同步 (grill-with-docs)

追问的同时**同步更新文档**：
- 架构决策 → 创建 ADR (`10_adr/`)
- 接口变更 → 更新 `AGENTS.md` 中的源码结构
- 运维流程 → 更新 `03_operations/runbooks/`
- 新规则发现 → 添加到 `rules/`

**不先实现后补文档**（永远不会补）。文档是实现的一部分。

## 反模式

❌ 用户说"加个指标" → 不问就加
✅ 用户说"加个指标" → 追问：哪个？参数？窗口？衍生特征需要吗？会影响 X.shape 吗？
